MCP Connectors for PMs

How Claude reaches out to your company's systems — and when to wire one up, use an approved one, or leave it alone. Built for PMs getting enterprise Claude access.

What MCP Actually Is

MCP (Model Context Protocol) is a plug-in system that lets Claude talk to external services — Jira, GitHub, Google Drive, Snowflake, Slack, Confluence, your internal tools. Each service exposes an "MCP server" with a set of actions Claude can call as tools.

Think of it as a driver pack for Claude. Without the Jira MCP connector, Claude doesn't know Jira exists. With it, Claude gets tools like jira_search_issues and jira_update_issue_status — just like how it has Read and Bash built in.

The three connection styles you'll encounter

Style How it works Typical example
Stdio (local) Claude spawns a local process on your laptop A Python script wrapping a REST API you run locally
HTTP / SSE (remote) Claude talks to a server over HTTPS An internal MCP your company hosts for a metric catalog
Hosted OAuth connector Vendor-managed; you click "Authorize" The official Atlassian / GitHub / Google Drive connectors

For a PM on Day 1: you'll almost always be in OAuth-connector territory. Stdio comes up only if a teammate ships a local wrapper. Self-hosted HTTP MCPs are usually built and deployed by your platform team.

Three Layers at Your Company

MCP configuration lives at three scopes. Know which layer you operate at before touching anything.

Enterprise admin

approved catalog

Your platform / IT team maintains a vetted list of MCP servers. This is how your company prevents unvetted connectors from exfiltrating data. Find out who owns the list and what's on it.

Project

.mcp.json

A repo-level file declaring which MCP servers the project needs. Teammates approve them when they first open Claude Code in the folder.

Personal

~/.claude/

Your own machine only. Add via claude mcp add or user settings. Personal productivity connectors no one else should see.

Own a project? → Decide which MCPs belong in .mcp.json.

Personal productivity? → User-level — don't pollute the team config.

Company-wide policy? → That's an enterprise-admin conversation, not a PM setup task.

Day-1 Playbook

What to actually do in your first week of enterprise access.

1Ask your admin for the approved connector list

Save it. That list is your toolbox. Most PMs are surprised how many connectors their company has already whitelisted.

2Connect the 2–3 you'll actually use

For a payments PM that's likely Jira, Confluence, and your Snowflake / BI connector if one exists. Resist the urge to connect everything.

3Test each one in a sandbox session

Ask something simple like "list my assigned Jira tickets this sprint". You want to know each connector works before you build workflows on top of it.

4Don't chase every connector

MCPs cost context budget. Ten connected servers means every prompt sees ten tool menus. Keep your stack to what you actively use.

Building Your Own MCP

The honest answer has tiers. Here's what it actually takes.

Tier Example Effort
Toy — one tool, wraps one REST endpoint get_weather(city) → JSON 1–2 hrs for a dev · a weekend for a motivated PM
Useful — a few tools, one service, error handling Wrap your internal feature-flag API: list / read / toggle 1 day for a dev · 2–4 days for a PM with an LLM helper
Production — many tools, auth, pagination, tests Full internal ticketing system with 20 actions 1–2 weeks; partner with an engineer
Polished platform — multi-tenant, observability, versioned Company-wide Snowflake MCP for all analysts Multiple engineer-weeks; don't solo this

What an MCP mechanically is

A program (any language) that speaks the Model Context Protocol — a JSON-RPC dialect — and exposes two things:

  1. Tool definitions — a list of callable actions with JSON schemas for their inputs.
  2. Tool handlers — the code that runs when Claude calls a tool.

Anthropic ships SDKs in Python and TypeScript that handle the protocol plumbing for you. You just write the tools. A functioning stdio MCP can be ~30 lines of Python.

The realistic PM scenario isn't hand-writing a production MCP. It's (1) writing the PRD for an internal MCP your engineers build, (2) prototyping a toy on your laptop to prove a concept, or (3) evaluating community MCPs before your team adopts one. All three are high-leverage PM work; solo-coding a production MCP usually isn't.

Gotchas Every PM Should Know

These aren't theoretical. They're the ones that actually bite teams adopting MCP.

Gotcha Severity Why it matters
Prompt injection via tool output High If your MCP returns data from Jira and someone wrote malicious instructions in a comment, those become text Claude might follow. A real risk, not a theoretical one.
Scope creep Medium Easy to add 30 tools to one MCP. Every tool is context budget. Fewer, richer tools beat many narrow ones.
Auth boundary High A local MCP runs with your credentials. If Claude is tricked into calling a destructive action, it runs as you. Restrict scopes at the connector level.
Data egress High Remote MCPs send data to an external server. Every new remote server needs intentional security review, not a one-click approval.
Versioning drift Medium Tool schemas change; saved workflows break silently. Pin versions where the server supports it.

The "Should I Build It" Decision

Three questions, in order. If you answer yes at any step, stop.

1Does this already exist as an approved connector?

  • →YesUse it don't rebuild

2Is this a one-person wrapper around a simple REST API?

  • →YesBuild a toy pitch the productionised version once proven

3Is this a system the whole company depends on?

  • →YesPlatform-engineering ask not a PM solo project

The enterprise-specific advice: when your access arrives, the most valuable thing you can do in week 1 isn't build anything. It's map the MCP surface — what connectors are enabled, what data each reaches, who owns each, and what's on the roadmap. That document alone makes you one of the more Claude-fluent PMs in your org, because most people start by poking at features instead of understanding the boundaries.
