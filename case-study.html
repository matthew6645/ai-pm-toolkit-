Why I built it

My company is rolling out Claude Code to engineering, and PMs are next. I had two options: wait for the rollout and learn it in flight, or learn it now by building something I'd actually use. I picked the second.

The brief I gave myself: ship a small set of focused PM tools that I'd reach for in my own work — not a "look what AI can do" demo, but something I could put in front of another PM and have them say "I'd actually use this."

How I worked with Claude Code

I treated each tool like a small product, not a prompt. Same loop every time:

  • Spec first. A short brief — what's the input, what's the output, what's the failure mode. Even one paragraph forces clarity.
  • Branch per change. Every tool, every guide, every animation tweak got its own PR. No mixing concerns.
  • Vercel preview before merge. If I couldn't click through it on a real URL, I didn't merge it.
  • Ship fast, edit later. A live preview I can react to is more useful than a perfect spec I'm still arguing with myself about.

Three product calls I'm proud of

Streaming over batched responses

Every tool renders Claude's output as it arrives, not after the full response. PMs iterate in real time — waiting 20 seconds for a wall of text breaks flow. The 50ms render debounce was a later add to keep the rendering smooth.

Conclusion-first decks

The Deck Builder forces consulting-style structure: lead with the recommendation, defend it after. Most PM decks build up to the ask. That's wrong for executive review, and it's the easiest thing to fix with a structured prompt.

No framework, no lock-in

The whole site is vanilla HTML/CSS/JS plus one Vercel serverless function. No React, no build step, no node_modules in the repo. Anyone can fork it, read it end-to-end in an afternoon, and run it themselves. Frameworks would have been faster to start, slower to share.

Three things I'd do differently

  • Set up a shared component pattern earlier. Each tool reinvented its own form, output panel, and copy button. By tool 3 I was repeating CSS I'd already written. A small components file from day 1 would have saved hours.
  • Add analytics from the start. I have no idea which tool gets used most, where people drop off, or whether the guides actually get read. Without data, my prioritization is guesswork.
  • Treat copy as a deliverable, not a placeholder. I shipped tool descriptions written in generic "AI tool for X" voice and only fixed them later. Voice is part of the product, not polish.

What I learned about working with AI as a PM

  • Specs matter more, not less. A vague prompt produces vague output, fast. The leverage is in writing the brief — the implementation is cheap.
  • Code review is the new design review. Reading what Claude wrote — and asking for changes — is where the product judgment happens. Skipping that step ships AI-flavored slop.
  • Iteration speed beats first-draft quality. Five rounds on a small change beats one round on a perfect change. Vercel previews on every PR made that loop fast enough to actually use.
