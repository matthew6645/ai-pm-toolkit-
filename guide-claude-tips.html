Claude Tips for PMs

A practical decision framework for getting the most out of Claude Code — built for product managers preparing for enterprise rollout.

The Four Questions

Before setting up any Claude behaviour, run through these four questions in order. They determine the right mechanism every time.

1Does this need to run every time, or only sometimes?

  • →Every prompt, automaticallyCLAUDE.md or Hook
  • →Only when you say soSlash command
  • →When the task matches a patternSubagent (main agent decides when)

2Do I need a hard guarantee, or is "the model usually does this" enough?

  • →Hard guaranteeHook
  • →Usually enoughCLAUDE.md

3Does this task need its own workspace?

  • →Yes — lots of intermediate output, or a different roleSubagent
  • →NoKeep it in the main conversation

4Is this personal to me, or for my team?

  • →Personal~/.claude/ (user-level)
  • →Team / project<project>/.claude/ (committed to the repo)

Worked Scenarios

Real situations mapped to the right mechanism — and why.

Scenario Best mechanism Why
Always audit a CSV before computing a metric CLAUDE.md rule + data-auditor subagent The rule belongs everywhere in the project (CLAUDE.md); the audit work is heavy and role-specialised (subagent)
One-click review of a PRD draft Slash command /prd-review Manual, predictable, shareable with other PMs via the repo
Three reviewers check every PR — security, metrics impact, rollback plan Three subagents, dispatched in parallel Isolation and parallelism is exactly what subagents are for
Every session, load our team's current metric definitions SessionStart hook that fetches an internal doc and injects into context External system, must happen every session, enforcement matters
No one on the team commits payment amounts without the _cents suffix PreToolUse hook on git commit Hard rule, compliance-relevant — hook beats "hope the model remembers"
Personal habit of asking Claude to summarise the day's git changes User-level slash command ~/.claude/commands/daily.md Personal, manual, doesn't belong in the repo
The whole org must follow our data-handling policy in every project Shared CLAUDE.md template + shared subagent files, distributed by the platform team Enterprise convention — belongs at project level, versioned

The Enterprise Scoping Question

The same mechanism can live at three scopes. Knowing which is which is critical.

User-level

~/.claude/

Your personal settings, invisible to teammates. Private recipes and personal preferences.

Project-level

<repo>/.claude/

Committed to the repo — every teammate gets them automatically. This is where team conventions live.

Enterprise-level

managed centrally

Deployed by your platform or DevEx team via a shared settings template. Applies across every project.

Should this apply to everyone? → Project-level CLAUDE.md or shared subagent.

Should teammates feel this rule? If no → User-level.

Does this need to apply outside my team? → That's a conversation with your platform team about enterprise defaults, not something you set up personally.
